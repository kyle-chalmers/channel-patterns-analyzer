    The step's content is a copy-into-response checklist (per Anthropic Skill best-practices cited in 02-RESEARCH.md and to mitigate Pitfall 3 — checklist skip under context pressure). The recipe must instruct the analyzer to literally copy the checklist into its working response and tick each item as it walks through the draft.

    **Required checklist items** (the full set; each item maps to a Phase 2 success criterion or a CLAUDE.md / CONTEXT.md rule):

    Structural checks (REPORT-01 / D-11 / D-12):
    - [ ] Six sections present in order: Data Health, Headline, What is working, What is not working, Patterns worth watching, Open questions.
    - [ ] Every section heading renders. Sections with no findings carry an explicit `Nothing material to report this week.` line OR a stale-table disclaimer (D-11).
    - [ ] Any section that would have drawn from a stale table (per Data Health) carries a one-line disclaimer naming the table and staleness (D-12). When multiple findings would disclaim the same table, the disclaimers are collapsed to one per section (RESEARCH.md Pitfall 5).

    Age-control checks (ANALYSIS-01 / ANALYSIS-04):
    - [ ] No video with `days_since_published < 14` appears in "What is working" top-performer claims.
    - [ ] Cross-age comparisons use a first-30-day window OR are explicitly labeled as a `views_per_day_since_publish_proxy` per the SQL file's own header comment.
    - [ ] Trending claims gated on at least 14 days of data per video in the comparison set.

    Confidence-label checks (ANALYSIS-03 / REPORT-02):
    - [ ] Every pattern claim ends with `(label, n=N)` parenthetical OR appears in a table with `Confidence` and `n` columns (D-07).
    - [ ] Each `n` cited matches the comparison set the claim actually drew from (RESEARCH.md Pitfall 2: tutorial-only claim cites tutorial-only `n`, not channel-wide `n`).
    - [ ] Labels match CLAUDE.md § "Small samples get hedged" thresholds: `< 5` → `low confidence`, `5 to 10` → `moderate confidence`, `>= 10` → `standard confidence`.

    Voice checks (REPORT-03):
    - [ ] No em dashes (U+2014) anywhere in the draft.
    - [ ] No en dashes (U+2013) used as punctuation.
    - [ ] None of the banned vocabulary from CLAUDE.md § "Voice" appears. The project list referenced authoritatively (per A8 in RESEARCH.md and CONTEXT.md "specifics"): delve, leverage, robust, seamless, navigate, underscore, showcase, tapestry, realm, multifaceted, transformative, testament to. The checklist item references the section title; the section title's list is authoritative even if it evolves.
    - [ ] No formulaic openers or closers: no `Great news!`, `Great question!`, `Overall,`, `In conclusion,`, `Ultimately,`, no contrastive `Not X, but Y` reframes, no opening transitions like `Moreover,`, `Furthermore,`, `Additionally,`.

    Prior-report citation checks (D-08 / D-09):
    - [ ] No prior-report citations in prose. Banned phrases: `"as we said last week"`, `"as noted previously"`, `"the prior report"`, `"this continues the trend we observed"`.
    - [ ] Multi-week claims (if any) stand on their own without requiring the reader to have seen a prior report (D-09 example: `"For the third consecutive week, X has held."` is allowed).

    Provenance check (CLAUDE.md § "Verification & Evidence" / global rule):
    - [ ] Numbers cited in the draft are present in the underlying `runs/{run_date}/queries/*.json` files. Spot-check three randomly chosen claims by grepping the value in the queries directory.

    Audit-trail recording (D-01 Layer 2 + Claude's Discretion):
    - For each PASSED check, record the check identifier (one of the snake-case names listed below) in `summary.json.voice_audit.checks_passed`.
    - For each FAILED check, fix the violation inline in the draft, then record a structured entry in `summary.json.voice_audit.fixes_applied` with shape `{"section": "<section name>", "fix": "<one-line description>"}`. Per RESEARCH.md "Extended summary.json schema" example.

    **Canonical check identifiers** (for `checks_passed`):
    - `six_sections_in_order`
    - `empty_sections_render_with_explicit_body`
    - `stale_table_disclaimers_present`
    - `age_control_enforced`
    - `cross_age_window_labeled`
    - `trending_claims_have_minimum_age`
    - `confidence_labels_present`
    - `confidence_n_matches_comparison_set`
    - `confidence_thresholds_correct`
    - `no_em_dashes`
    - `no_en_dashes_as_punctuation`
    - `no_banned_vocab`
    - `no_formulaic_openers`
    - `no_prior_report_citation`
    - `multi_week_claims_self_contained`
    - `numbers_match_underlying_query_results`

    **Step gate (RESEARCH.md Pitfall 3 mitigation):**

If a check cannot be ticked (data unavailable, edge case the checklist doesn't cover), record the reason in `summary.json.voice_audit.fixes_applied` with `section: "(audit)"` and `fix: "could not verify {check_identifier}: {reason}"` — then proceed only if the reason is genuinely outside the checklist's scope."
