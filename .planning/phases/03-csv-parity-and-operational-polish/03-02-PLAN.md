## Plan Structure Rationale

This plan owns the **CSV query-engine slice** (Slice 1 of MVP Phase 3, sub-task B). It creates the brand-new `scripts/csv_query.py` helper that Plan 03-03 (the recipe's CSV branch) will invoke. Per RESEARCH.md § "CSV Engine Recommendation" the stdlib helper is locked over DuckDB for three reasons: BigQuery SQL doesn't round-trip to DuckDB without translation; the query surface is small (4 queries, stable); and `.planning/PROJECT.md` § "Constraints" prefers "shell + SQL + Claude" over a Python SDK.

`files_modified` is `scripts/csv_query.py` (new) + `CHANGELOG.md`. Plan 03-01 also touches `CHANGELOG.md`, so within Wave 1 these two plans serialize on that file. The serialization is harmless: each plan appends a separate H2-dated section and CHANGELOG appends are commutative at the section level. Wave 1 wave assignment stays correct because the BLOCKING dependency is the recipe edit (Plan 03-03), not the CHANGELOG.

**MVP slice check:** After this plan ships, the helper exists and is unit-callable, but the analyzer's `/run-analyzer` recipe doesn't invoke it yet (Plan 03-03 wires that). The analyzer remains end-to-end runnable in BigQuery-mode throughout this plan's lifetime; CSV-mode end-to-end runnable lands only after Plan 03-03.

**Rejected alternative:** DuckDB-reads-CSV. Documented in `03-RESEARCH.md` § "CSV Engine Recommendation"; tradeoff is rejected because (a) BigQuery's `CURRENT_DATE("America/Phoenix")` doesn't round-trip to DuckDB's tz-aware date functions without a dialect-translation layer, and (b) adding a 30MB package violates `.planning/PROJECT.md` "no new Python packages without justification."

    3. Compute `days_stale = (_today_phoenix() - latest_snapshot).days` (when `latest_snapshot` is not None).
