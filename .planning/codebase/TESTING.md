## Mocking

Not applicable. The closest mocking-shaped construct is the **CSV fallback mode** (`DATA_SOURCE=csv` in `.env`), which substitutes `sample_data/*.csv` for live BigQuery queries. This is a runtime fixture, not a unit-test mock.

**What `sample_data/` does:**
- Lets viewers and CI-shaped checks run the analyzer end-to-end without BigQuery credentials.
- Schema matches the four BigQuery tables exactly (`scripts/csv_fallback_loader.py:21-41` defines 13 full-length videos + 3 shorts).
- Seeded `random.Random(42)`, `random.Random(43)`, `random.Random(44)` for deterministic output across regenerations (`scripts/csv_fallback_loader.py:69,89,116`).
- Includes one deliberately-young video (`VID013`, 3 days old) to exercise the age-control rule in `BUSINESS_RULES.md` §3.

**Pattern for future fixture extension:**
```python
def generate_<table_name>(snapshot: date) -> list[dict]:
    rows: list[dict] = []
    rng = random.Random(<distinct_seed>)
    for vid, ... in SAMPLE_VIDEOS:
        rows.append({...})
    return rows
```
Use a distinct seed per table to keep cross-table values independent but deterministic.

## Test Types

**Unit Tests:** None.

**Integration Tests:** None in the conventional sense. The closest equivalent is a manual analyzer run against either:
1. Live BigQuery (`DATA_SOURCE=bigquery`), or
2. Generated CSVs (`DATA_SOURCE=csv` with `sample_data/` populated).

**End-to-End Tests:** The weekly scheduled run **is** the production E2E exercise. Every Monday at 9am Phoenix time (see `docs/schedule.md`), the analyzer:
1. Queries BigQuery.
2. Drafts a report.
3. Publishes to Notion.
4. Writes `reports/{run_date}.md` and `runs/{run_date}/`.

The `runs/{run_date}/summary.json` file is the test-result record. Per `runs/README.md:59`, `summary.json` must be written on success **and** failure, with `errors: [...]` populated on failure.

**Schema-drift / data-contract checks:** Encoded directly in SQL rather than in a test runner:
- `sql/04_data_health_check.sql` flags any of the four tables stale beyond `CURRENT_DATE() - 3` (per `BUSINESS_RULES.md` §3).
- `sql/01_latest_snapshot_overview.sql:15-20` uses `LEAST(MAX, MAX)` to refuse to join across mismatched snapshot dates.

## Common Patterns

**Verification Without a Test Runner:**

The repo's verification model is **artifact-driven**, not assertion-driven:

1. **Pre-run check:** `bq query --use_legacy_sql=false "SELECT 1"` to confirm BigQuery auth (`docs/maintenance.md:33`).
2. **In-run data health:** Every report opens with a "Data Health" section listing latest snapshot per table and flagging staleness. This is mandatory per `CLAUDE.md` § Report structure.
3. **Post-run audit:** `runs/{run_date}/queries/*.json` captures the raw rows the analyzer reasoned over. When a published claim is later questioned, the audit folder shows whether the rows supported the claim or whether the interpretation was at fault (`docs/runbook.md:79-90`).

**Async Testing:** Not applicable.

**Error Testing:** Failure modes are documented as procedural playbooks in `docs/runbook.md`, not as test cases. Each section names the symptom, the recovery steps, and where to record it (typically `CHANGELOG.md`). New failure modes are added to the runbook as part of the fix.

**Confidence Calibration (a verification pattern unique to this repo):**
- Sample-size thresholds defined in `CLAUDE.md` § Small samples get hedged: <5 videos = low confidence, 5–10 = moderate, 10+ = standard.
- Age-control: exclude videos with `days_since_published < 14` from "top performers" and pattern claims (`BUSINESS_RULES.md` and `sql/03_age_controlled_performance.sql:35`).
- These are enforced by the analyzer at reasoning time, not at test time. There is no automated check that catches a claim violating them — the audit trail in `runs/` exists so a human can catch it after the fact.

## Adding Tests Later (Guidance)

If automated tests become warranted, the natural seams are:

1. **SQL smoke tests:** Run each `sql/*.sql` against `sample_data/` (via a BigQuery sandbox or a SQLite shim) and assert row counts / column presence. Place under `tests/sql/`.
2. **Fixture regeneration determinism:** Assert that `python scripts/csv_fallback_loader.py` produces byte-identical CSVs given a fixed `date.today()` mock. Place under `tests/scripts/`.
3. **Schema-drift guard:** Compare live BigQuery schemas against an expected manifest derived from `BUSINESS_RULES.md` §4. Failure means update the rules or fix the upstream pipeline.

Until then, the audit trail in `runs/` plus the runbook in `docs/runbook.md` is the verification system.
