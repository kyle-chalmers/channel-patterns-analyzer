## Directory Purposes

**`sql/`:**
- Purpose: BigQuery query templates the analyzer runs each week.
- Contains: Numbered `.sql` files; numeric prefix encodes run order.
- Key files: `01_latest_snapshot_overview.sql`, `02_top_full_length_videos.sql`, `03_age_controlled_performance.sql`, `04_data_health_check.sql`.

**`scripts/`:**
- Purpose: One-off utility scripts that aren't part of the weekly run.
- Contains: Python stdlib-only utilities.
- Key files: `csv_fallback_loader.py`.

**`reports/`:**
- Purpose: Human-readable weekly outputs. Also the analyzer's calibration memory across runs.
- Contains: One markdown file per run date.
- Key files: `README.md` (explains the naming convention and how the analyzer reads prior reports).

**`runs/`:**
- Purpose: Machine-readable audit trail per run.
- Contains: One subfolder per run date with `summary.json`, `queries/*.json`, and `report.md`.
- Key files: `README.md` (defines the `summary.json` schema).

**`docs/`:**
- Purpose: Operator manual. Only read when something needs to change or has broken.
- Contains: Failure-mode playbook, maintenance guide, scheduling notes.
- Key files: `runbook.md`, `maintenance.md`, `schedule.md`.

**`images/`:**
- Purpose: Source and rendered diagrams used in `README.md`.
- Contains: Excalidraw source plus exported PNG.
- Key files: `diagram.excalidraw` (4-layer architecture), `diagram-decision-tree.excalidraw`.

**`sample_data/`:**
- Purpose: CSV fallback data for the no-BigQuery path.
- Contains: Four CSVs matching the BQ schema.
- Note: Gitignored. Generated by `scripts/csv_fallback_loader.py`.

**`.planning/`:**
- Purpose: GSD working artifacts (plans, codebase maps).
- Contains: `codebase/` subdirectory with this and other map files.
- Note: Gitignored per global convention.

**`.internal/`:**
- Purpose: Kyle's personal config and recording materials.
- Contains: Owner config, recording notes.
- Note: Gitignored. Do not read these from agent sessions other than `OWNER_CONFIG.md`.

## Key File Locations

**Entry Points:**
- `CLAUDE.md`: Loaded automatically when Claude Code starts in this repo. Defines the entire analyzer.
- `scripts/csv_fallback_loader.py`: Manual Python entry for sample-data generation.

**Configuration:**
- `.env.example`: Template for `BQ_PROJECT`, `BQ_DATASET`, `YOUTUBE_CHANNEL_ID`, `NOTION_REPORT_PAGE_ID`, `ANALYSIS_LOOKBACK_DAYS`, `MIN_VIDEO_AGE_DAYS`, `SCHEDULE_TIMEZONE`, `DATA_SOURCE`.
- `.env`: Real values; gitignored.

**Core Logic (prose, not code):**
- `CLAUDE.md`: Voice, age control, sample-size hedging, report structure, run protocol.
- `BUSINESS_RULES.md`: Fiscal calendar, data-health expectations, per-table grain, join-key contract.

**Core Logic (SQL):**
- `sql/01_latest_snapshot_overview.sql`: Health-check overview joining `video_metadata` and `daily_video_stats`.
- `sql/02_top_full_length_videos.sql`: Raw top-N ranking, age applied downstream by the analyzer.
- `sql/03_age_controlled_performance.sql`: Implements the `>= 14 days` filter and views-per-day proxy.
- `sql/04_data_health_check.sql`: Staleness check per table for the Data Health section.

**Persistence:**
- `reports/{YYYY-MM-DD}.md`: Weekly report. Filename = run date, not snapshot date.
- `runs/{YYYY-MM-DD}/summary.json`: Per-run audit metadata.
- `runs/{YYYY-MM-DD}/queries/*.json`: Raw query result dumps.

**Documentation:**
- `README.md`: Public-facing setup and Context Engineering framing.
- `PROMPTS.md`: The 10-prompt sequence that builds the analyzer.
- `CHANGELOG.md`: Material-change audit.
- `docs/`: Operator manual.

- `.claude/skills/write-notion-report/SKILL.md`: Notion publish skill.

## Naming Conventions

**Files:**
- Top-level operating files: `UPPERCASE.md` (`README.md`, `CLAUDE.md`, `BUSINESS_RULES.md`, `CHANGELOG.md`, `PROMPTS.md`, `LICENSE`).
- SQL: `NN_snake_case.sql` where `NN` is a zero-padded two-digit run-order prefix (`01_`, `02_`, ...). New queries take the next prefix per `docs/maintenance.md`.
- Python: `snake_case.py`.
- Date-stamped artifacts: `YYYY-MM-DD` using the **run date**, not the snapshot date. Same-day re-runs append `-2`, `-3`.

**Directories:**
- Top-level: lowercase, single word where possible (`sql/`, `scripts/`, `reports/`, `runs/`, `docs/`, `images/`).
- Dotted-private: `.planning/`, `.internal/`, `.claude/` ‚Äî all gitignored.
- Run subfolders: `runs/{YYYY-MM-DD}/`, `runs/{YYYY-MM-DD}-2/` for retries.

**SQL identifiers:**
- Bare dataset references: `` `youtube_analytics.<table>` `` with no project prefix. The `bq` CLI resolves the project from active `gcloud` config. Forks rename via templating `$BQ_DATASET`.
- Snake_case column and table names matching the upstream `youtube-bigquery-pipeline` schema.

**Markdown headers:**
- Section dividers in SQL comments use the box-drawing pattern `-- ‚îÄ‚îÄ‚îÄ Title ‚îÄ‚îÄ‚îÄ`.
- CHANGELOG entries are dated H2 headings (`## 2026-05-24`) with bulleted changes underneath.

## Where to Add New Code

**New SQL query:**
- Primary file: `sql/NN_descriptive_name.sql` using the next available numeric prefix.
- Header: include a comment referencing the relevant `BUSINESS_RULES.md` section and the dataset-name convention (`youtube_analytics` placeholder).
- Register: list the file in the analyzer's query manifest in `CLAUDE.md` (or wherever the run manifest lives).
- Log: add a one-line `CHANGELOG.md` entry per `docs/maintenance.md`.

**New business rule:**
- Primary file: `BUSINESS_RULES.md`, in the smallest possible diff.
- Log: add a `CHANGELOG.md` entry stating before/after values for any threshold moved.
- Never change `BUSINESS_RULES.md` ¬ß1 (fiscal-year anchor) without discussing ‚Äî it would retroactively reinterpret every prior report.

**New analyzer behavior (voice, hedging, section order):**
- Primary file: `CLAUDE.md`.
- Log: `CHANGELOG.md` entry; note whether the change would have altered recent reports.

**New utility script:**
- Primary file: `scripts/<purpose>.py`.
- Default to Python stdlib only (matches `csv_fallback_loader.py` and avoids polluting `requirements.txt`).
- If new deps are required, add them to `requirements.txt` with a comment explaining when they're needed.

**New documentation page:**
- Operator manual: `docs/<topic>.md`. Update `docs/README.md`'s table.
- Public-facing: extend `README.md` rather than creating a new top-level doc.

**New failure-mode handler:**
- Add a section to `docs/runbook.md` describing symptom, fix, and recording note. Per `docs/maintenance.md`, this is part of the fix, not a follow-up.

**New report (manual / one-off):**
- Land at `reports/{YYYY-MM-DD}.md` and `runs/{YYYY-MM-DD}/` like any other run. If it should not enter the archive (e.g., test of a broken query), say so up front or delete before committing.
